# AGENTS.md

## Mermaid 작성 규칙

- Mermaid 차트를 작성할 때는 **문법 안전성 우선**으로 작성한다.
- 가능하면 `graph TB` 또는 `graph LR`와 기본 사각 노드(`[]`)만 사용한다.
- 노드 ID는 영문/숫자(`A1`, `B2`)처럼 단순하게 사용한다.
- 라벨은 따옴표(`"..."`)로 감싸고, 복잡한 특수문자 사용을 최소화한다.
- 파서 호환성이 불명확한 고급 문법(특수 노드 모양, 과도한 HTML, 초기화 블록)은 기본적으로 피한다.

## Mermaid 자가 교정 절차

- 차트 작성 후 문법 오류가 발생하면, 에러를 그대로 두지 말고 즉시 수정한다.
- 수정 순서:
1. `flowchart`를 `graph`로 단순화한다.
2. 시작/종료 특수 노드(예: `([..])`)를 기본 노드(`[".."]`)로 바꾼다.
3. 결정 노드/특수 스타일을 제거하고 기본 노드 + 라벨 화살표만 남긴다.
4. HTML 태그(`br`)나 특수기호를 줄여 한 줄 텍스트로 단순화한다.
5. 그래도 실패하면 가장 작은 동작 가능한 차트부터 다시 확장한다.

## 작업 원칙

- "머메이드 차트가 렌더러마다 다르게 깨질 수 있다"는 점을 항상 가정한다.
- 정교함보다 **실제로 파싱되는 문법**을 우선한다.
- 오류 제보를 받으면 즉시 문법 호환 버전으로 재작성한다.

## 프로그래머스 작성 규칙

- 프로그래머스식으로 작성할 때는 `class Solution` 내부의 `fun solution(...)` 안에서 모든 함수/변수/로직을 처리한다.
- 특수 네이밍 규칙(예: DFS/BFS 네이밍 규칙)을 제외하면 변수명은 한국어를 우선 사용한다.
- 문제 상단 주석에는 입력/정답을 한눈에 확인할 수 있는 "예시 표"를 가능하면 함께 적는다.
- 예시 표를 넣었으면 표 아래에 각 값의 의미를 짧게 설명한다.
- 특히 `return` 열은 각 원소/형식이 무엇을 뜻하는지 반드시 적는다.
- 예시 표 형식 예:
  - 예시
  - | N | picture | return |
  - |---|---|---|
  - | 5 | ["RRRBB","GGBBB","BBBRR","BBRRR","RRRRR"] | [4, 3] |

## DFS/BFS 네이밍 규칙

- DFS/BFS 문제는 가능하면 `dfs()`, `bfs()` 함수로 탐색 로직을 분리한다.
- 그래프/자료구조 변수명은 `graph`로 고정하지 말고 문제 맥락에 맞는 이름을 사용한다.
- 예: 인접 리스트면 `인접목록`, 격자면 `지도`, 트리면 `트리`, 간선 모음이면 `간선목록`.
- 방향 벡터는 `dx`, `dy`(또는 `dw`, `dh`)를 사용한다.
- 좌표 변수명은 `x`, `y`를 사용한다.
- 다음 좌표 변수명은 `nx`, `ny`를 사용한다.
- 큐 변수명은 무조건 `큐`를 사용한다.
- 개수 변수명은 `xCnt`, `yCnt`를 사용한다.
- 방문 배열 변수명은 `visited`를 사용한다.
- 방문 배열은 `BooleanArray`를 쓰지 않고 항상 `MutableList<Boolean>` 타입을 사용한다.
- 예: 1차원 `MutableList(n) { false }`, 2차원 `MutableList(yCnt) { MutableList(xCnt) { false } }`.

## 엘레베이터 템플릿 생성 프롬프트

아래 프롬프트를 그대로 복사해서 사용한다.  
문제 정보만 바꾸면 `엘레베이터.kt` 스타일로 코드/스키마를 함께 생성하도록 설계되어 있다.

```text
너는 Kotlin 알고리즘 코드 정리 도우미다.
아래 문제 정보를 바탕으로 "엘레베이터.kt 스타일"로 결과를 만들어라.

[문제 정보]
- 문제명: {문제명}
- 코드 파일 경로: {코드_경로}
- 스키마 파일 경로: {스키마_경로}
- 입력 정의: {입력_정의}
- 출력 정의: {출력_정의}
- 제약조건: {제약조건}
- 예시 입력/출력: {예시}
- 핵심 아이디어 메모(선택): {아이디어_메모}

[필수 요구사항]
1) 코드는 프로그래머스식으로 작성:
   - class Solution
   - fun solution(...): ...
   - main/readln/print 사용 금지

2) 주석 스타일은 엘레베이터.kt와 동일한 밀도/형식:
   - 상단 문제 요약 블록
   - KDoc에 "핵심 아이디어", "데이터 구조", "시뮬레이션(또는 탐색) 순서", "주석 템플릿 규칙"
   - 본문 단계 주석 [S0], [S1]..., [S_END]
   - 상태 변수 주석 [V0], [V1]...
   - 상태 변수 선언/핵심 갱신 라인에 [V*]를 정확히 표기
   - 알고리즘 핵심 분기/확장 라인에 [S4]를 반드시 표기

3) 스키마 마크다운 파일({스키마_경로}) 생성:
   - Mermaid는 반드시 graph TB 또는 graph LR
   - 기본 사각 노드만 사용
   - S 노드와 V 노드를 모두 포함
   - 조건 분기 화살표에는 YES/NO 라벨 명시
   - 복잡한 문법/HTML/특수 노드 사용 금지

4) 코드-스키마 싱크 규칙:
   - 코드에 등장한 [S*], [S_END], [V*] 집합 == 스키마의 S/V 집합
   - 누락/불일치가 있으면 결과 제출 전에 스스로 수정
   - 스키마의 단계 의미가 코드 실행 순서와 일치해야 함

5) 결과 품질:
   - 설명은 풍부하되 장황하지 않게
   - 단계/상태 이름은 일관성 유지
   - 초심자가 코드와 스키마를 나란히 보고 바로 추적 가능해야 함

[출력 형식]
- 수정/생성된 코드 파일 전체
- 수정/생성된 스키마 md 파일 전체
- 마지막에 "S/V 싱크 체크 결과"를 짧게 요약
```
